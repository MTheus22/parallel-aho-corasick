Visão geral

As 3 fases de uma execução

Toda execução do laboratório passa exatamente por estas fases em sequência. As fases são estanques: a busca nunca modifica o autômato, e o merge nunca reprocessa o texto. Isso é o que torna o paralelismo na fase de busca trivialmente livre de locks.

Fase 1
Construção

Carrega padrões, monta trie, executa BFS para calcular as funções de transição, falha e dict_suffix. Resultado: autômato imutável.

Sequencial padrão Paralelo opt-in (Idea 4)
Fase 2
Busca (Search)

Cada worker lê o autômato e seu chunk de texto, emite matches na sua lista privada. Nenhum dado é compartilhado entre workers em escrita.

1 thread (baseline) N threads (paralelismo)
Fase 3
Merge

O master thread concatena as listas thread-locais em ordem de thread_id após o pthread_join. Representa <1% do tempo total.

Sempre sequencial Irrelevante p/ otimizar
✅
As fases 1 e 2 são completamente independentes e combináveis. Você pode usar o build paralelo (Idea 4) com qualquer searcher, ou o build sequencial com qualquer searcher. O autômato produzido é bit-a-bit idêntico nas duas variantes — verificado por automata_byte_equal() em make test. Veja a seção "Combinabilidade" abaixo.
Fase 1

Construção do autômato

Converte o dicionário de padrões em um DFA determinístico completo que permite transitar com um único load indexado por byte de entrada. A construção é um gargalo real apenas para dicionários muito grandes (ET ~44k regras); para dicionários menores domina a fase de busca.

Etapa 1: inserção na trie
Sempre sequencial · ac_automaton_build()
Percorre os bytes de cada padrão criando nós na trie (raw_child). Custo O(total_pattern_length) — tipicamente <5% do custo total do build. Não foi paralelizado: a disputa por slots de filho tornaria necessário atomics com custo/benefício desfavorável.
Resultado
Trie pronta em raw_child[]
Etapa 2: BFS — goto + fail + dict_suffix
Sequencial padrão · Paralelo com AC_BUILD_PARALLEL=1
BFS a partir da raiz preenche goto_tbl, fail[] e dict_suffix[]. Em cada nível d, cada estado depende apenas de estados em d-1 — isso é a invariante que torna o nível embarrassingly parallel. Custo dominante: O(num_states × 256).
Ideia atacada
Idea 4 — BFS paralela (✅ feito)
Etapa 3: flat output table
Sequencial · Idea 5 · Sempre ativo · e78045c
Um pass extra após a BFS pré-computa, para cada estado, a lista contígua de pattern_ids que AC emitiria ao chegar ali. Elimina dois níveis de pointer-chasing na emissão. Searchers antigos ignoram os novos campos; searchers novos (_flat) os leem.
Ideia atacada
Idea 5 — flat outputs (✅ feito) Todos os searchers herdam
💡
Idea 4 — BFS Paralela: ativa via AC_BUILD_PARALLEL=1 AC_BUILD_THREADS=N. Usa uma BFS nível-síncrona com workers por nível e um atomic_fetch_add para reservar slots no frontier de próximo nível. É o único atomic de toda a implementação e só dispara durante o build, nunca no hot loop de busca. Após pthread_join ao final de cada nível, todos os writes do nível anterior ficam visíveis (happens-before garantido pelo join).
Fase 2

Busca (Search) — 17 searchers registrados

Cada searcher é um arquivo .c independente que se registra via __attribute__((constructor)). O Makefile os descobre por glob. Todos precisam concordar com o sequential em make test nos thread counts {1,2,3,4,7,8}.

Baseline

sequential
Single-thread · Referência de correção
Loop simples: state = goto_tbl[state*256+c] por byte. Emissão via chain-walk (own_out_head + dict_suffix). É o oráculo contra o qual todos os outros são validados.

Resultados medidos: Snort 207 MB/s · ET-32 103,7 MB/s (enron_x8)

Eixo 1: Paralelismo de dados (particionamento do texto)

O texto é dividido em chunks, cada thread recebe um. O autômato é compartilhado em leitura. O gargalo residual é o cache blowout em dicionários grandes.

pthread_chunked
Chunks estáticos homogêneos · v1 original
Divide o texto em N partes iguais. Cada worker faz warm-up de max_pattern_len-1 bytes antes do seu chunk para garantir o estado correto do DFA na fronteira. Emissão com chain-walk.
Ataca
Paralelismo básico de busca T=12: 4,13× speedup (Snort)
pthread_chunked_v2
Split warm-up/owned loops · cache-pad em worker_t
Divide o loop em duas fases: warm-up (sem emissão) e owned (com emissão). Elimina o branch if (i < core_start) do hot loop. Padding do worker_t para 64 bytes evita false-sharing entre workers.
Ataca
False-sharing Branch no hot loop
pthread_chunked_v3 ⭐
Topology-aware · Chunks ponderados por frequência
Lê a frequência de cada core no sysfs (cpufreq) e distribui chunks proporcionalmente: P-cores recebem ~142 MB, E-cores ~107 MB. Minimiza a ociosidade na barreira do pthread_join.

Melhor resultado medido: 735 MB/s · 4,78× speedup (Snort, T=12)
Ataca
Desbalanceamento P/E-core Ociosidade na barreira
pthread_dynamic
Dispatch dinâmico via contador atômico
Divide o texto em 4N tarefas menores. Workers pegam a próxima tarefa com um atomic_fetch_add. Adapta-se a flutuações térmicas em tempo real, mas tem overhead do atomic no dispatch.
Ataca
Flutuação de carga dinâmica
pthread_prefetch
Software prefetch do stream de texto
Adiciona __builtin_prefetch(text + Δ) no hot loop (Δ ≈ 512B), instruindo o hardware a carregar linhas de cache futuras enquanto processa as atuais. Não ajuda com o autômato (acesso imprevisível), mas reduz stalls de leitura do texto.

Latência de leitura do texto

Eixo 3: Paralelismo de dicionário (Idea 1)

Em vez de dividir o texto, divide o dicionário. K sub-autômatos disjuntos, cada worker varre o texto inteiro com seu sub-autômato K vezes menor. Política retida: prefix-bucket (única que vence o sequencial no regime alvo).

pattern_sharded_prefix ⭐
Idea 1 · Multi-thread · 2026-05-21 · Política vencedora
Bucket pelo primeiro byte: shard = pat[0] % K. Padrões com a mesma letra inicial caem no mesmo shard; sub-autômatos têm raízes densas e hot path coerente em cache.

Melhor resultado: 121,40 MB/s · 1,21× speedup (Snort full + enron, K=8) — única política que vence o sequential. Empiricamente satura em K=2.
Ataca
Cache locality por shard Cache blowout (parcial)
📊
Conclusão honesta da idea 1: abordagem regime-dependente. Para dicionários grandes (Snort full ≥ 56 MB), a política prefix entrega 1,2× sobre o sequencial; round-robin e LPT regridem. Para dicionários pequenos (Snort-100 cabe em L2), nenhuma política vence — o "second-scan cache hit" não compensa o tráfego K× de texto. Mesmo o melhor caso (prefix, 1,21×) é ~3× menor que pthread_chunked_flat (3,64× contra o mesmo baseline) — paralelismo de texto continua sendo o eixo dominante. A composição 2-D (K × N) foi implementada na idea 6 abaixo — e falsifica a hipótese de que componha; veja o eixo 4.

Eixo 4: Composição 2-D K × N (Idea 6 — hipótese falsificada)

Combina o eixo 3 (sharding por dicionário, idea 1) com o eixo 1 (chunking por texto, idea 5): K sub-autômatos × N chunks de texto = T workers. O searcher constrói K=2 sub-autômatos pela política prefix e despacha N=T/K workers por shard. Hipótese registrada em idea_6.md: K=2 × N=6 (T=12) deveria bater pthread_chunked_flat em dicionários grandes trocando banda de texto por cache locality.

pthread_2d_sharded_chunked
Idea 6 · Multi-thread (T = K × N) · 2026-05-22
Constrói K (default 2, override via AC_2D_K) sub-autômatos com bucketing pelo primeiro byte. Cada worker (k, n) faz scan do chunk n com sub-autômato k; warm-up por shard usa sub_aut[k].max_pattern_len - 1. Emissão via tabela achatada (idea 5) + remapeamento pid_remap[k] para o pid global.

⚠ Hipótese falsificada: em todos os dicionários e thread counts medidos, o 2-D perde 10–27% para pthread_chunked_flat.

Snort full + simplewiki T=12: 356,60 MB/s (3,97×) vs 488,28 MB/s (5,43×) do chunked_flat — ratio 0,73×.
ET-32 + simplewiki T=12: 166,80 MB/s (3,40×) vs 195,23 MB/s (3,99×) do chunked_flat — ratio 0,85×.
Ataca
Cache blowout + banda de texto (composição) Perdeu p/ chunked_flat em todos os regimes
📉
Por que caiu: em K=2, o sub-autômato (~258 MiB para ET-32) ainda é 21× a L3 — não sai do regime DRAM-bound. Mas o tráfego de texto agora é 2× maior (cada worker re-lê seu chunk pelo seu sub-autômato). A banda de DRAM do i5-1235U (~25 GB/s) satura antes que o ganho parcial de cache locality apareça. K-sweep em T=12 confirma: vazão decresce monotonicamente de K=1 (177,46 MB/s) para K=12 (71,60 MB/s).

Valor científico apesar do "perde": negativo é resultado. A composição 2-D fecha a grade `(1-D K, 1-D N, 2-D K×N)` e dá ao leitor do TCC todos os pontos para inferir que paralelismo de texto domina paralelismo de dicionário neste hardware. Em servidor com banda > 200 GB/s e L3 > 128 MiB, a relação pode reverter.

Eixo 5: Composição ortogonal com flat (Idea 7) — hipótese parcialmente refutada

Combina a otimização v3 (afinidade topológica + chunks ponderados por frequência) com a emissão flat (idea 5). A hipótese, registrada na idea-card F1 da síntese do TCC, é que o ataque é ortogonal ao flat layout. Sweep multi-regime (4 combinações patterns × corpus, 2026-05-22) refuta parcialmente: pthread_chunked_flat continua sendo o melhor em 3 dos 4 regimes; F1 (v3_flat) ganha modestamente apenas em Snort + Enron (+2,6 %). A composição prefetch + flat (idea 8) foi testada e removida do escopo (negativo qualificado: regredia em 4/4 regimes — a lição já é carregada pelo pthread_prefetch não-flat).

pthread_chunked_v3_flat
Idea 5 ⊕ 7 · Multi-thread · 2026-05-22 · Ganho regime-específico
Composição do pthread_chunked_v3 (topology-aware affinity + freq-weighted chunks via sysfs) com a emissão flat. Workers pinados em líderes de núcleo físico antes de irmãos SMT; bytes proporcionais a cpuinfo_max_freq; emissão sem chain-walk.

Resultado multi-regime (T=12): empata em R1 (Snort + SimpleWiki, −0,7 %); ganha +2,6 % em R2 (Snort + Enron, regime DRAM-bound); regride 26,3 % em R3 (Snort-100 cache-friendly, overhead de leitura de sysfs domina); regride 9,5 % em R4 (ET-32).

⚠ Útil apenas para o regime DRAM-bound moderado. Em cache-friendly, o overhead da sondagem de topologia/freq elimina o ganho.
Ataca
+2,6 % apenas em R2 (Snort + Enron) Regride 26 % em R3 (cache-friendly)
📊
Lição das composições ortogonais — hipótese parcialmente refutada: a idea 5 (flat output table) é um multiplicador estrutural que já está próximo do ótimo dado o hardware (i5-1235U) e os workloads testados. Compor com afinidade topológica (v3) só recupera 2,6 % em Snort + Enron — o único regime onde o gargalo é parcialmente desbalanceamento P/E-core. Compor com prefetch de texto regride em todos os regimes porque o HW prefetcher do x86_64 já saturou o caminho. O resultado científico válido é empírico: existem ataques ortogonais ao flat layout (idea 5), mas o teto de ganho neste hardware é ~3 %; demais composições simples regridem. A próxima frente real de ganho não é composição em CPU desktop, mas mudança de hardware (servidor com mais L3 e DRAM bandwidth — F4 da síntese).
Composição

Combinabilidade: Build × Search

Construção e busca são fases estanques. O autômato é o único ponto de contato — e ele é idêntico nas duas variantes de build. Portanto qualquer par (build, searcher) é válido.

✅ Sim — são combináveis

AC_BUILD_PARALLEL=1 + qualquer searcher funciona porque ac_automaton_build_par() produz um struct bit-idêntico a ac_automaton_build(). O searcher não sabe nem se importa com qual função construiu o autômato — ele só lê campos estáticos.


Exemplos válidos:
sequential build + pthread_chunked_v3 ✓
parallel build (T=8) + pthread_chunked_flat ✓
parallel build (T=8) + pattern_sharded_prefix ✓

⚠ Atenção nas medições

O ganho do build paralelo é no tempo de construção, não no throughput de busca. O CLI reporta os dois separados: # build time: e bench_run. Para a dissertação, sempre reporte os dois — especialmente em dicionários grandes onde o build pode ser 10-30% do tempo total em uso real (IDS que recarrega regras).


Pattern sharding (Idea 1) constrói K sub-autômatos internamente. Nesses sub-autômatos, ac_automaton_build() será chamada K vezes — portanto o build paralelo (Idea 4) pode ser chamado internamente pelo sharding também, compondo os dois.

Build Searcher Válido? O que mede
ac_automaton_build Qualquer dos 13 registrados ✅ Sempre Apenas ganho de busca
ac_automaton_build_par(T) Qualquer dos 13 registrados ✅ Sempre Ganho de build + ganho de busca
Qualquer um pattern_sharded_prefix ✅ Válido K sub-builds internos + busca em paralelo de dicionário
Qualquer um pthread_2d_sharded_chunked ✅ Válido K sub-builds + N chunks de texto. Override de K via AC_2D_K

Fase B — Impacto do footprint do autômato (enron · 1,32 GiB · T=12)

🔥
Este é o problema central do TCC: throughput cai 78,7% conforme o dicionário cresce de 1,3 MB para 515 MB. A Idea 1 (pattern sharding) foi a abordagem desenhada para atacar diretamente esse gargalo — e foi parcialmente bem-sucedida: a política prefix entrega 1,21× sobre o sequencial em Snort full (~56 MB), mas não recupera o regime cache-friendly do Snort-100 (1.838 MB/s) — fica em 121 MB/s. O standalone tem teto; a saída é a composição 2-D com chunking (ver seção Idea 1 abaixo).

Referência

Glossário de termos técnicos

Aho–Corasick
Algoritmo de busca multi-padrão que constrói um DFA a partir de um conjunto de strings e escaneia o texto com uma única passagem, detectando todas as ocorrências de todos os padrões em O(|texto| + |matches|). É a base de ferramentas IDS como Snort e Suricata.
DFA — Autômato Finito Determinístico
Estrutura que, dado um estado atual e um byte de entrada, determina exatamente um próximo estado. No AC, a função de transição é goto_tbl[state * 256 + byte]. "Determinístico" = sem ambiguidade, sem backtracking — cada byte custa exatamente um load indexado.
Trie
Árvore de prefixos onde cada caminho da raiz a uma folha representa um padrão. Construída primeiro; depois transformada em DFA via BFS que resolve as transições de falha.
Fail link (link de falha)
Para cada estado u, fail[u] aponta para o estado que representa o sufixo próprio mais longo de u que é também prefixo de algum padrão. Usado internamente na BFS de construção para propagar transições ausentes. Após o build, o goto_tbl já incorpora os fail links — o search nunca precisa seguir fail[] explicitamente.
dict_suffix
dict_suffix[u] aponta para o ancestral mais próximo na cadeia de fail que tem saídas próprias (i.e., é terminal de algum padrão). Permite emitir todos os matches de uma posição sem percorrer a cadeia de fail byte a byte — um salto direto ao próximo estado emissor.
goto_tbl / função de transição plana
Array int32_t[num_states × 256] que armazena o DFA completo. Para qualquer estado e qualquer byte de entrada, goto_tbl[state*256+byte] dá o próximo estado — direto, sem fail links residuais. É a escolha de layout que torna o hot loop um único load indexado por byte.
Warm-up (aquecimento do DFA)
Nos searchers por chunks, antes de emitir matches no seu intervalo próprio, cada worker processa os max_pattern_len − 1 bytes anteriores ao seu chunk sem emitir nada. Isso leva o DFA a um estado consistente com o que uma varredura sequencial teria na fronteira. Sem warm-up, padrões que cruzam fronteiras de chunk seriam perdidos.
Overlap (sobreposição)
Os max_pattern_len − 1 bytes no final do chunk do worker anterior que o worker atual também lê (como warm-up). "Overlap" não significa que os matches são duplicados — os matches dentro da região de warm-up são descartados pelo worker que os processou durante o warm-up; apenas o worker "dono" daquela região emite.
Ownership de matches
Regra que determina qual worker reporta um match: o worker i reporta apenas matches com end_pos ∈ [core_start_i, core_end_i). Matches encontrados durante o warm-up (antes de core_start) são silenciosamente descartados — eles serão reportados pelo worker do chunk anterior.
Cache blowout / cache miss
Quando a estrutura de dados usada pela CPU excede a capacidade dos caches (L1, L2, L3) e precisa buscar dados na RAM. O autômato ET-32 (~515 MB) é 42× maior que a L3 do i5-1235U (12 MB). Cada transição de estado resulta em um cache miss, e a CPU fica esperando dados da DRAM a cada byte — é por isso que o throughput cai 78,7%.
Memory bandwidth (largura de banda de memória)
Taxa máxima de transferência de dados entre CPU e RAM. É o gargalo absoluto no regime DRAM-bound: não importa quantas threads adicionais sejam criadas, todas competem pelo mesmo barramento. Em IDS com dicionários grandes, é o limite físico que impede speedup linear.
Thread-local matches
Cada worker mantém sua própria ac_match_list_t que somente ele escreve. Não há mutex, não há atomic. O master thread coleta todas após pthread_join. Isso é possível porque os ranges de ownership são disjuntos — não há duplicação de matches.
False sharing
Quando duas threads escrevem em variáveis diferentes que estão na mesma linha de cache (64 bytes). O hardware invalida a linha inteira a cada escrita, fazendo as threads "brigarem" pelo cache line mesmo sem conflito lógico. Mitigado em pthread_chunked_v2 com padding da struct worker_t para 64 bytes.
Speedup
Razão entre o tempo do baseline sequencial e o tempo da variante: speedup = T_seq / T_par. Speedup linear (ideal) = número de threads. Na prática, é sublinear por overheads de sincronização, memory bandwidth e arquitetura híbrida P/E-core.
Level-synchronous BFS (BFS nível-síncrona)
Variant da BFS que processa todo o nível d antes de avançar para d+1. No AC, cada estado em d depende apenas de estados em d-1 — portanto todos os estados do mesmo nível podem ser processados em paralelo sem comunicação entre si. A barreira entre níveis é um pthread_join.
BFS frontier (fronteira da BFS)
O conjunto de estados atualmente em processamento na BFS nível-síncrona. frontier_curr[] contém os estados do nível atual; workers descobrem filhos reais e os escrevem em frontier_next[] via atomic_fetch_add para reservar slots sem colisão.
Flat output table (tabela de saídas achatada)
Para cada estado s, um array contíguo de pattern_ids que seriam emitidos ao chegar em s. Substitui dois níveis de listas ligadas (own_out_head + outputs e dict_suffix). Elimina pointer-chasing dependente na emissão, especialmente relevante quando o autômato não cabe em cache (loads aleatórios em listas ligadas = muitos cache misses).
δ² (delta-dois) — multi-step transition
Tabela pré-computada onde δ²[state][byte_pair] codifica a transição de dois bytes de uma vez: goto_tbl[goto_tbl[state*256+a]*256+b]. Reduz a cadeia de loads dependentes pela metade. Footprint: num_states × 65536 × 4 bytes — só viável com estados < 3300.
Pattern sharding (divisão do dicionário)
Em vez de dividir o texto entre workers, divide o dicionário. Cada worker recebe um subconjunto dos padrões, constrói seu próprio sub-autômato (K vezes menor) e varre o texto inteiro. O ganho é que sub-autômatos menores cabem em L2/L3 quando o DFA unificado não cabe.
pthread_join / barreira de sincronização
pthread_join(tid) bloqueia até a thread terminar e garante que todas as escritas desta thread são visíveis ao chamador (happens-before). É a única barreira de memória no modelo de paralelismo do laboratório — não há mutexes, semáforos ou spinlocks no hot loop.
AC_UNLIKELY / branch prediction
Macro que expande para __builtin_expect(x, 0), indicando ao compilador que a condição (geralmente um match) é rara. O preditor de branch da CPU pode otimizar o código do caminho raro sem penalizar o caminho comum (byte sem match, que é a grande maioria).
ThreadSanitizer (TSan) / data race
Ferramenta de instrumentação de tempo de execução que detecta data races — acessos concorrentes ao mesmo endereço de memória onde pelo menos um é escrita, sem sincronização. make tsan compila com -fsanitize=thread. Se o autômato é verdadeiramente read-only e os matches são thread-local, o TSan deve ficar silencioso.
Software prefetch
__builtin_prefetch(addr) instrui a CPU a começar a carregar uma linha de cache antes que ela seja necessária, escondendo a latência. Útil para o stream de texto (acesso sequencial e previsível). Não ajuda para o autômato (acesso dirigido pelo estado atual — imprevisível).
P-core / E-core (Intel Alder Lake)
O i5-1235U tem 2 cores de Performance (P, mais rápidos, com Hyperthreading) e 8 cores de Eficiência (E, mais lentos, sem HT). Com chunks iguais, os E-cores ditam a barreira final — os P-cores ficam ociosos esperando. pthread_chunked_v3 distribui chunks proporcionais à frequência para reduzir essa assimetria.
